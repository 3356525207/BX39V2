# Verification 6.1 — Bluetooth Commands Accepted While UART1 Is Silent (Bench Build)

**Feature: e0-bypass-bt-testing, Requirement 5**
**Validates: Requirement 5.1**

## Method

There is **no host test harness** and **no ARM toolchain** available in this workspace
(verified: `arm-none-eabi-gcc`, `armcc`, `armclang`, `make`, and `cmake` are all absent from
PATH; no host test files exist under the tree). The Bluetooth command path is tightly coupled to
hardware (`UART2_SendData`, `UART1_GetTemp`, `RCV315_*`, `g_ms_tick`, interrupt-context parsing),
so this task is executed as a **documented state-transition verification** by static analysis of
the actual, unmodified source, exactly as the design's Testing Strategy §3 prescribes for the
"Commands accepted while UART1 silent" example/integration case.

No source logic was modified for this verification. No compiler or test run was fabricated.

## Bench build definition

The bench-test build defines `TEST_BYPASS_COMM_ERR`, wired in the MDK-ARM project
(`MDK-ARM/LedBlink.uvprojx`) preprocessor defines:

```
N32G003, USE_STDPERIPH_DRIVER, TEST_BYPASS_COMM_ERR
```

## The command gate and why the bypass matters

The gate that rejects commands under a communication error lives in `BT_Matrix()`
(`myapp/bt_transparent.c`). Its `SYS_STATE_COMM_ERR` case rejects everything except `stop`:

```c
case SYS_STATE_COMM_ERR:           /* error (安全扣/通信) — 无法经蓝牙清除 */
    switch (cmd) {
    case BT_CMD_STOP:     return MX_IDEMPOTENT;
    default:              return MX_REJECT;
    }
```

`BT_Matrix()` is the sole state×command decision point used by both dispatchers
(`BT_DispatchControl()` for reg 0x00 run-control / mode, `BT_DispatchSetSpeed()` for reg 0x01
setSpeed). The action it returns maps directly to the acknowledgement:

| `BT_Matrix()` action | Effect | `ack` |
|----------------------|--------|-------|
| `MX_ACCEPT` | `SYS_RUN_HandleBTCtrl(cmd)` runs the transition (or `BT_ApplySetSpeed`) | `status:"ok"` |
| `MX_IDEMPOTENT` | no transition | `status:"ok"` |
| `MX_BUSY` | no transition | `status:"warning"`, `error.code:"DEVICE_BUSY"` |
| `MX_REJECT` | no transition | `status:"warning"`, `error.code:"COMMAND_REJECTED"` |

**Chain established in prior tasks:** Task 4.2 verified that under `TEST_BYPASS_COMM_ERR` the
2 s UART1 ACK timeout never sets `sys_state = SYS_STATE_COMM_ERR` (the sole timeout entry point
into that state is excluded from compilation). Therefore, with no motor controller connected and
UART1 permanently silent, `sys_state` remains at the **real operating state**
(`STANDBY`/`RUNNING`/`PAUSED`/…). Consequently `BT_Matrix(sys_state, cmd)` is evaluated against
that real state and **never** against `SYS_STATE_COMM_ERR` — so it returns exactly the same
action, and `SYS_RUN_HandleBTCtrl()` performs exactly the same transition, as a **production
build sitting in the same non-error state**.

This is the crux of Requirement 5.1: the bypass does not add or alter any command-handling
logic; it merely keeps `sys_state` out of `COMM_ERR`, restoring the identical command path a
production build would take when not in an error state.

## Frame format (from `BT_ParseByte()`)

```
0xAA 0x55  <cmd>  <reg>  <len>  <data…len bytes>  0x55
```
- `cmd`: `0x02` = write, `0x03` = read.
- reg `0x00` → run-control / mode; `data[0]` = command code.
- reg `0x01` → setSpeed; `data[0]` = speed×10 (valid `0x0A`–`0x26`).

Run-control command codes (reg 0x00 `data[0]`) as found in `BT_CmdToAckOf()` /
`SYS_RUN_HandleBTCtrl()`:

| Code | Command | `ackOf` |
|------|---------|---------|
| 0x00 | stop    | stop    |
| 0x01 | start   | start   |
| 0x02 | pause   | pause   |
| 0x03 | resume  | resume  |
| 0x04 | mode1   | mode1   |
| 0x05 | mode2   | mode2   |
| 0x06 | mode3   | mode3   |
| 0x07 | mode4   | mode4   |

## Per-command walkthrough (bench build, UART1 silent, `sys_state` = real operating state)

For each command the parse path is `BT_ParseByte × N → BT_ProcessFrame → dispatcher →
BT_Matrix(sys_state, cmd) → (SYS_RUN_HandleBTCtrl) → BT_RequestAck`. Because `sys_state` is the
real state (never `COMM_ERR`), every result below is **identical to a production build in the
same state**.

### stop (0x00) — frame `AA 55 02 00 01 00 55`
- `BT_DispatchControl(0x00)` → `BT_Matrix(state, 0x00)`.
- `STANDBY/STOPPED` → `MX_IDEMPOTENT` → ack `stop`/ok, no transition.
- `RUNNING/PAUSED/COUNTDOWN_START/COUNTDOWN_RESUME` → `MX_ACCEPT` →
  `SYS_RUN_HandleBTCtrl(0x00)` → `SYS_STATE_STOPPING`, `Beep_Tem = BEEP_LONG`, ack `stop`/ok.
- `VIBRATION` → `MX_ACCEPT` → `SYS_STATE_STANDBY`, ack `stop`/ok.
- Under production-in-COMM_ERR this would be `MX_IDEMPOTENT` (ok, no transition); under bypass it
  is evaluated against the real state as above — matching a production build not in error. ✔

### start (0x01) — frame `AA 55 02 00 01 01 55`
- Thermal gate `BT_ThermalNotNormal()` checked first (unchanged by bypass; on a silent bench
  `UART1_GetTemp()` reads a non-hot value and `sys_state != ESTOP`, so gate passes).
- `BT_Matrix(state, 0x01)`:
  - `STANDBY/STOPPED` → `MX_ACCEPT` → `SYS_RUN_HandleBTCtrl(0x01)` →
    `SYS_STATE_COUNTDOWN_START`, `Beep_Tem = BEEP_SHORT`, ack `start`/ok.
  - `RUNNING/COUNTDOWN_*` → `MX_IDEMPOTENT` → ack `start`/ok.
  - `PAUSED` → `MX_REJECT` → ack `start`/warning `COMMAND_REJECTED` (use resume).
- In production-COMM_ERR start returns `MX_REJECT`; under bypass it now `ACCEPT`s from idle
  exactly as a non-error production build. ✔

### pause (0x02) — frame `AA 55 02 00 01 02 55`
- `BT_Matrix(state, 0x02)`:
  - `COUNTDOWN_START/RUNNING/PAUSED/COUNTDOWN_RESUME` per matrix
    (`ACCEPT`/`IDEMPOTENT`) → running/starting/resuming → `SYS_STATE_PAUSED`,
    `Beep_Tem = BEEP_LONG`, ack `pause`/ok; paused → idempotent ok.
  - `STANDBY/STOPPED/VIBRATION` → `MX_REJECT` → ack `pause`/warning `COMMAND_REJECTED`.
- Matches non-error production; never blocked by COMM_ERR under bypass. ✔

### resume (0x03) — frame `AA 55 02 00 01 03 55`
- Thermal gate checked (same as start).
- `BT_Matrix(state, 0x03)`:
  - `PAUSED` → `MX_ACCEPT` → `SYS_RUN_HandleBTCtrl(0x03)` → `SYS_STATE_COUNTDOWN_RESUME`,
    `Beep_Tem = BEEP_SHORT`, ack `resume`/ok.
  - `RUNNING/COUNTDOWN_*` → `MX_IDEMPOTENT` → ack `resume`/ok.
  - `STANDBY/STOPPED/VIBRATION` → `MX_REJECT` → ack `resume`/warning `COMMAND_REJECTED`.
- Matches non-error production. ✔

### setSpeed (reg 0x01) — frame `AA 55 02 01 01 <spd> 55`  (e.g. `<spd>=0x14` → 2.0)
- `BT_ProcessFrame()` range-checks `spd ∈ [0x0A,0x26]` (else `COMMAND_REJECTED`), then
  `BT_DispatchSetSpeed(spd)`.
- Thermal gate checked first, then `BT_Matrix(state, BT_CMD_SETSPEED=0x80)`:
  - `COUNTDOWN_START/RUNNING/PAUSED/COUNTDOWN_RESUME` → `MX_ACCEPT` → `BT_ApplySetSpeed`
    (clamp 1.0–3.8, set `g_target_speed`, and if `RUNNING` call `RCV315_SetSpeed`), ack
    `setSpeed`/ok.
  - `STANDBY/STOPPED/VIBRATION/STOPPING` → `MX_REJECT`/`MX_BUSY` per matrix → warning ack.
- The setSpeed sentinel (`0x80`) is not a reg-0 control code; it participates in the matrix only.
  Under bypass the state fed to the matrix is the real state, so setSpeed behaves as in non-error
  production. ✔

### mode 0x04–0x07 (reg 0x00) — frame e.g. `AA 55 02 00 01 04 55`
- Handled by the early branch in `BT_DispatchControl()`: accepted **only** in
  `STANDBY`/`STOPPED`/`VIBRATION`:
  - Accepted → `SYS_RUN_HandleBTCtrl(cmd)`: from idle/stopped sets vib level `cmd-0x03` and
    enters `SYS_STATE_VIBRATION` (`Beep_Tem = BEEP_SHORT`); from vibration just switches level.
    ack `mode{1..4}`/ok.
  - Any other state → ack `mode{n}`/warning `COMMAND_REJECTED`.
- This branch does not consult `BT_Matrix`, but it does read `sys_state`. Under production-COMM_ERR
  `sys_state` would be `COMM_ERR` (not in the accept set) → all modes rejected. Under bypass
  `sys_state` is the real state, so modes are accepted from idle exactly as non-error production. ✔

## Equivalence table (bench-bypass vs. production)

| Command | Bench build, UART1 silent | Production build, same non-error state | Equal? |
|---------|---------------------------|----------------------------------------|--------|
| stop 0x00    | matrix vs real state → transition/idempotent + ack | identical | ✔ |
| start 0x01   | idle→COUNTDOWN_START + ok / idempotent / reject | identical | ✔ |
| pause 0x02   | running→PAUSED + ok / reject | identical | ✔ |
| resume 0x03  | paused→COUNTDOWN_RESUME + ok / reject | identical | ✔ |
| setSpeed r01 | clamp+set target (+apply if running) + ok / reject | identical | ✔ |
| mode 0x04–07 | idle→VIBRATION set level + ok / reject | identical | ✔ |

The **only** difference between the two columns in the "UART1 silent" scenario is that a
production build would already have transitioned to `SYS_STATE_COMM_ERR` (E0) after 2 s and would
therefore reject all of the above except an idempotent `stop`. The bypass removes that single
divergence; once `sys_state` is held at the real non-error state, the command outcomes are
byte-for-byte the same production code path (same `BT_Matrix`, same `SYS_RUN_HandleBTCtrl`, same
`BT_RequestAck`).

## Conclusion

Requirement 5.1 is satisfied in the bench build by static analysis of the actual, unmodified
source. Every listed UART2 command (stop 0x00, start 0x01, pause 0x02, resume 0x03, setSpeed
reg 0x01, mode 0x04–0x07) is parsed by `BT_ParseByte()` and dispatched through the identical
`BT_Matrix()` / `SYS_RUN_HandleBTCtrl()` logic used in production. Because the bypass keeps
`sys_state` out of `SYS_STATE_COMM_ERR` while UART1 is silent (established in Task 4.2), the
command gate is evaluated against the real operating state, producing the same transitions and
`ack` responses as a production build in an equivalent non-error state. The command-handling
source is not touched by the bypass — no `TEST_BYPASS_COMM_ERR` reference exists in
`bt_transparent.c` or in `SYS_RUN_HandleBTCtrl()`.

  `BT_DispatchSetSpeed()`, `BT_ApplySetSpeed()`, `BT_RequestAck()` ack mapping.
