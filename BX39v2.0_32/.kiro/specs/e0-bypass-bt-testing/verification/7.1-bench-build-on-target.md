## 1. Purpose

Prove, on the BX39 v2.0 board with **no motor controller connected on UART1**, that the bench
build (`TEST_BYPASS_COMM_ERR` defined):

- Never enters the E0 communication-timeout error and never emits the 800 ms COMM_ERR beep (Req 2).
- Still processes the full Bluetooth (UART2) command set with normal acks/telemetry (Req 5).
- Still enforces the E17 safety-clip protection on PA4 (Req 3).
- Still enforces the E2 overheat emergency-stop protection (Req 4).

### Build & flash the bench target

1. Open `MDK-ARM/LedBlink.uvprojx` in Keil µVision.
2. In the target selector, choose **`N32G003_BenchTest`**.
3. Confirm Options for Target → C/C++ → Define contains `TEST_BYPASS_COMM_ERR`.
4. Rebuild (F7). Output is `Objects_BenchTest\LedBlink_BenchTest.axf`.
5. Flash to the board (Load / F8) with the debug probe.

## 4. Bluetooth frame format (UART2)

From `BT_ParseByte()` in `myapp/bt_transparent.c`:

```
0xAA 0x55  <cmd>  <reg>  <len>  <data…len bytes>  0x55
```
- `cmd`: `0x02` = write, `0x03` = read.
- `reg 0x00` → run-control / mode; `data[0]` = command code.
- `reg 0x01` → setSpeed; `data[0]` = speed × 10 (valid `0x0A`–`0x26`).

Run-control command codes (reg 0x00 `data[0]`):

| Code | Command | Example write frame |
|------|---------|---------------------|
| 0x00 | stop    | `AA 55 02 00 01 00 55` |
| 0x01 | start   | `AA 55 02 00 01 01 55` |
| 0x02 | pause   | `AA 55 02 00 01 02 55` |
| 0x03 | resume  | `AA 55 02 00 01 03 55` |
| 0x04 | mode1   | `AA 55 02 00 01 04 55` |
| 0x05 | mode2   | `AA 55 02 00 01 05 55` |
| 0x06 | mode3   | `AA 55 02 00 01 06 55` |
| 0x07 | mode4   | `AA 55 02 00 01 07 55` |
| setSpeed (reg 0x01) | e.g. 2.0 mph | `AA 55 02 01 01 14 55` (`0x14`=20 → 2.0) |

Acks are JSON-style responses with `status:"ok"` on accept/idempotent, or `status:"warning"`
with `error.code` (`COMMAND_REJECTED` / `DEVICE_BUSY`) on reject/busy.

## 5. Test procedure

Execute steps in order. Record PASS/FAIL and observations in the sign-off table (§6).

### Step 1 — E0 suppression (Req 2.1, 2.2)
1. With the bench build flashed and **no controller on UART1**, power up (or reset) the board.
2. Leave the board idle for **at least 5 seconds** (well past the 2 s ACK-timeout window).
3. **Expected:**
   - Display **never** shows `E0` at any point.
   - **No** periodic 800 ms COMM_ERR beep occurs.
   - Board remains in its normal operating (standby) state and stays there indefinitely — repeat
     observation after ~30 s to confirm E0 never appears (Req 2.2).
   - _Validates 2.1 (state unchanged on timeout), 2.2 (out of E0 indefinitely)._

### Step 2 — Bluetooth command path (Req 5.1, 5.2)
Keep UART1 silent throughout. Connect the BLE client, then send each frame and observe the ack
plus the display/telemetry. Because the firmware is held out of COMM_ERR, outcomes must match a
production build in the same (non-error) state.

1. **start** `AA 55 02 00 01 01 55` from standby → ack `start`/ok; board enters countdown→running;
   speed telemetry begins updating.
2. **setSpeed 2.0** `AA 55 02 01 01 14 55` while running → ack `setSpeed`/ok; target speed updates
   to 2.0 mph in telemetry.
3. **pause** `AA 55 02 00 01 02 55` while running → ack `pause`/ok; board enters paused.
4. **resume** `AA 55 02 00 01 03 55` while paused → ack `resume`/ok; board resumes (countdown→running).
5. **stop** `AA 55 02 00 01 00 55` while running → ack `stop`/ok; board stops → standby.
6. **mode1–mode4** `AA 55 02 00 01 04 55` … `07 55` from standby → ack `mode{n}`/ok; board enters
   the corresponding vibration mode/level.
7. **Expected:** every command is accepted/processed with the normal ack (no spurious
   `COMMAND_REJECTED` caused by COMM_ERR), and telemetry streams normally.
   - _Validates 5.1 (commands processed as in production), 5.2 (not blocked by COMM_ERR)._

> Note: `start`/`resume`/`setSpeed` pass the thermal cooling gate first; on a silent bench with no
> injected over-temp this gate is open, so these commands proceed normally.

### Step 3 — E17 safety-clip preserved (Req 3.1, 3.2, 3.3)
1. From a running (or standby) state, **assert PA4 high** (simulate safety-clip release) per P5.
2. **Expected:**
   - Firmware transitions to COMM_ERR with `g_safety_err = 1`.
   - Display renders **`E17`**.
   - On the UART1 TX line, **emergency-stop frames** are transmitted (`UART1_SendEmergencyStop()`).
     When PA4 returns low, the firmware transmits emergency-**release** frames
     (`UART1_SendEmergencyRelease()`) as part of the defined safety-clip recovery path.
   - _Validates 3.1 (COMM_ERR + g_safety_err=1), 3.2 (E17 shown), 3.3 (emergency stop/release TX)._
3. Restore PA4 low to exit the safety condition before the next step (the release path clears the
   error once acknowledged; on a bench without a controller, re-flash/reset if needed to return to
   a clean state).

### Step 4 — E2 overheat ESTOP preserved (Req 4.1, 4.2)
1. Using the UART1 injection fixture (P6), feed motor-temperature telemetry with a value
   **> 130 °C** and hold it continuously for **≥ 10 seconds** (the firmware requires 10 s of
   continuous confirmation before latching the level-4 overheat state — see `SYS_RUN_Process()`).
2. **Expected:**
   - Firmware transitions to `SYS_STATE_ESTOP`.
   - Display renders **`E2`**.
   - Speed is forced to zero and the long overheat alarm sounds.
   - _Validates 4.1 (>130 °C → ESTOP), 4.2 (E2 shown)._
3. Remove the over-temp injection and reset the board to return to a clean state.
