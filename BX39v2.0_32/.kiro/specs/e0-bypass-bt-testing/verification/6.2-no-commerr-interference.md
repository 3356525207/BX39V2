# Verification 6.2 — No COMM_ERR interference after the 2 s silent window (bench build)

`Feature: e0-bypass-bt-testing, Requirement 5`

**Validates: Requirements 5.2**

## Objective

Confirm that, under the bench build (`TEST_BYPASS_COMM_ERR` defined), after the 2 s UART1
silent window the Bluetooth command matrix is evaluated against the **real operating state**
(idle / running / paused / …) and **never** against `SYS_STATE_COMM_ERR` **as a result of the
ACK-timeout path**. (E17/E2 can still legitimately drive `COMM_ERR`/`ESTOP` — those are real
safety states, not timeout interference, and are out of scope for this criterion.)

## Method

No host unit-test harness (Unity/Ceedling/CMake) and no ARM toolchain exist in this environment
(searched: no `CMakeLists.txt`, no `unity.h`, no `test/` tree — same finding as verification
4.1). Per the design's Testing Strategy (Section 3, "No COMM_ERR interference"), this task is
executed as a **documented static data-flow / state-transition verification** — a trace through
the command-gate code proving the state fed into the matrix is the live `sys_state`, combined
with the timeout-suppression proof from task 4.2. No source logic was modified and no
compiler/test run was fabricated.

Source of truth for this trace:
- `myapp/bt_transparent.c` — `BT_Matrix()`, `BT_DispatchControl()`, `BT_DispatchSetSpeed()`
- `myapp/ctrl_tx.c` — `CTRL_TX_Process()` (the `#ifndef TEST_BYPASS_COMM_ERR` timeout guard)
- `myapp/SYS_RUN.c` — `SYS_RUN_HandleBTCtrl()` (accept-cell executor)

## Build configuration (bench)

`TEST_BYPASS_COMM_ERR` is **defined** (bench-test build). Consequently the ACK-timeout `if`
block in `CTRL_TX_Process()` is compiled **out**; only the `#else` comment remains. This is the
established result of task 4.2.

## The command matrix reads the live `sys_state`

The command gate never caches or snapshots a state value — it passes the global `sys_state`
directly into `BT_Matrix()` at the moment each frame is dispatched:

- `BT_DispatchControl()` (reg 0x00, start/stop/pause/resume):
  ```c
  act = BT_Matrix(sys_state, cmd);
  ```
- `BT_DispatchSetSpeed()` (reg 0x01, setSpeed):
  ```c
  act = BT_Matrix(sys_state, BT_CMD_SETSPEED);
  ```
- The mode branch (0x04–0x07) and the thermal gate (`BT_ThermalNotNormal()`) likewise read the
  live global `sys_state` at dispatch time.
- The accept-cell executor `SYS_RUN_HandleBTCtrl(cmd)` also `switch (sys_state)`es on the same
  live global.

`sys_state` is the single global (`extern uint8_t sys_state;`) shared by CTRL_TX and the BT
path. There is no separate copy. Therefore the matrix result is a pure function of **whatever
`sys_state` holds at the instant the command is processed**.

## Why that live state is never `COMM_ERR` via the timeout path (bench build)

`BT_Matrix()` contains a dedicated `case SYS_STATE_COMM_ERR:` that rejects everything except
`STOP` (idempotent). This is exactly the interference the criterion guards against: if a UART1
silent window pushed `sys_state` into `COMM_ERR`, every subsequent start/pause/resume/setSpeed
would be forced down the `MX_REJECT` cell regardless of what the user actually intended.

The **only** timeout writer of `SYS_STATE_COMM_ERR` is the guarded block in `CTRL_TX_Process()`:

```c
#ifndef TEST_BYPASS_COMM_ERR
    if (sys_state != SYS_STATE_COMM_ERR && (now - last_ack > 2000)) {
        sys_state = SYS_STATE_COMM_ERR;   /* E0 — compiled OUT under bypass */
        ...
    }
#else
    /* TEST_BYPASS_COMM_ERR: E0 suppressed; state left unchanged */
#endif
```

Under the bench build this block is absent from the translation unit (proved in 4.2). Hence, no
matter how long UART1 stays silent (`now - last_ack > 2000`, indefinitely), `sys_state` is left
untouched by the timeout logic and keeps whatever real operating value the state machine set
(`SYS_STATE_STANDBY`, `_RUNNING`, `_PAUSED`, `_COUNTDOWN_START`, `_COUNTDOWN_RESUME`,
`_STOPPING`, `_STOPPED`, `_VIBRATION`).

## Step-by-step trace: command after the 2 s silent window

Stimulus: bench build, no motor controller connected, UART1 silent for > 2000 ms, device in a
real operating state (take `SYS_STATE_RUNNING` as the representative case; the argument is
identical for idle/paused/etc.).

1. Time passes with `UART1_CheckAndClearAck()` returning `false`; `last_ack` is never refreshed,
   so `now - last_ack` grows past 2000 ms.
2. `CTRL_TX_Process()` runs each 200 ms. Safety-clip (PA4) is low ⇒ E17 block skipped; ACK
   branch skipped. The timeout block **does not exist** in this build ⇒ `sys_state` is **not**
   changed. `sys_state` remains `SYS_STATE_RUNNING`.
3. A BLE frame arrives on UART2; `BT_ParseByte()` assembles it and `BT_ProcessFrame()` routes it
   to `BT_DispatchControl()` / `BT_DispatchSetSpeed()`.
4. The dispatcher evaluates `BT_Matrix(sys_state, cmd)` with `sys_state == SYS_STATE_RUNNING` —
   the **real** operating state. It enters `case SYS_STATE_RUNNING:`, **not**
   `case SYS_STATE_COMM_ERR:`.
5. The returned action (e.g. `MX_ACCEPT` for stop/pause/setSpeed, `MX_IDEMPOTENT` for start) is
   the normal-operation cell, identical to what a production build in the same non-error state
   would return. Accept cells call `SYS_RUN_HandleBTCtrl(cmd)`, which again switches on the same
   live `SYS_STATE_RUNNING`.

At no point in this build can the ACK-timeout path route the matrix into `case
SYS_STATE_COMM_ERR:`.

### Assertions

| Assertion | Result | Basis |
|-----------|--------|-------|
| Matrix input is the live `sys_state`, not a cached value | ✅ | `BT_Matrix(sys_state, …)` called at dispatch time |
| Timeout path cannot set `sys_state = COMM_ERR` under bypass | ✅ | timeout `if` compiled out (`#ifndef`, task 4.2) |
| After 2 s silence, matrix evaluated against real operating state | ✅ | `sys_state` unchanged ⇒ `case SYS_STATE_RUNNING/…` reached |
| `case SYS_STATE_COMM_ERR:` never reached via timeout | ✅ | no timeout writer exists in this build |
| E17/E2 can still set COMM_ERR/ESTOP (legitimate safety) | ✅ (out of scope) | those writers are outside the guard; not timeout interference |

## Result

**PASS (by static data-flow / state-transition verification).** In the bench build
(`TEST_BYPASS_COMM_ERR` defined), the command matrix is always evaluated against the live global
`sys_state`, and the ACK-timeout path — the sole timeout writer of `SYS_STATE_COMM_ERR` — is
compiled out. Therefore, after the 2 s UART1 silent window the matrix is evaluated against the
real operating state (idle / running / paused / …) and is never routed into
`case SYS_STATE_COMM_ERR:` by the timeout. This satisfies Requirement 5.2. Legitimate safety
transitions to `COMM_ERR` (E17/PA4) and `ESTOP` (E2/overheat) remain fully active and are not
affected by this criterion.

Note: full dynamic confirmation on hardware is covered by task 7.1 (bench build end-to-end:
BLE start/stop/pause/resume/speed/mode accepted with normal acks while no controller is
connected).
