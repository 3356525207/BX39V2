# Verification 7.2 — Production build end-to-end (on-target runbook)

`Feature: e0-bypass-bt-testing, Requirement 6`

**Validates: Requirements 6.1, 6.2, 6.3**

> **ON-TARGET RUNBOOK — NOT executable in this environment.**
> This task is a physical hardware integration check on the BX39 v2.0 board (Nations
> N32G003 MCU). It requires the board, an ARM flashing tool, and direct observation of the
> 7-segment display and UART1 line. Automated execution is **not possible** in this
> workspace (no target hardware, no flasher, no display). This document is the procedure to
> be run by a technician on the bench. **No hardware results are recorded here** — the
> results/sign-off table below must be filled in by the person who runs the procedure.

## Purpose

Prove that the **production** firmware configuration retains the E0 (UART1 ACK-timeout)
protection: with no motor controller connected on UART1, the display must show **"E0"**
within ~2 s of power-on, the commanded speed must be forced to 0, and the downstream
emergency-stop command (`0xAA55`) must be transmitted continuously on UART1
(Requirements 6.1, 6.2, 6.3).

This is the counterpart to task 7.1 (bench build). Running both back-to-back on the same
board demonstrates that the two build configurations differ **only** in the E0 timeout path
and nowhere else.

## Build configuration under test (production)

| Item | Value |
|------|-------|
| Toolchain / project | MDK-ARM (Keil µVision), `.uvprojx` |
| Device target | N32G003 |
| `TEST_BYPASS_COMM_ERR` | **undefined** (in-source `#define` at top of `myapp/ctrl_tx.c` left commented; no project define adds it) |
| Output image | `LedBlink` (production build artifact) |

With the macro undefined, the `#ifndef TEST_BYPASS_COMM_ERR` branch in `CTRL_TX_Process()`
(`myapp/ctrl_tx.c`) — the real ACK-timeout block — is what compiles into the image.

## Preconditions

1. BX39 v2.0 board available and powered off, with the 7-segment display visible.
2. **No motor controller board connected on UART1** (this is what makes the ACK time out).
   Confirm the UART1 downstream connector is unplugged.
3. ARM flashing tool ready and connected to the board's SWD/debug header
   (e.g. Keil µVision + debug probe, or the equivalent Nations flashing utility).
4. (Optional but recommended) A logic analyzer or UART sniffer on the UART1 TX line to
   observe downstream frames. Emergency-stop frame is `0xAA 0x55` (`UART1_SendEmergencyStop()`).
5. Production image built from the current source with the N32G003 MDK-ARM target,
   `TEST_BYPASS_COMM_ERR` undefined, output artifact `LedBlink`.

## Procedure

Perform each step in order and record the observed result in the sign-off table.

1. **Select and build the production target.** In MDK-ARM, select the N32G003 target with
   `TEST_BYPASS_COMM_ERR` undefined. Build; confirm a clean build producing the `LedBlink`
   output image.
2. **Flash the production build** to the BX39 v2.0 board using the flashing tool. Confirm the
   flash/verify step reports success.
3. **Power on the board** (or issue a reset after flashing) with **no motor controller
   connected on UART1**. Start a stopwatch at power-on/reset.
4. **Observe the 7-segment display.** Within **~2 s** of power-on, the display SHALL show
   **"E0"** (rightmost two active digits render `E` then `0`). Record the elapsed time to
   first "E0". _(Req 6.1 → COMM_ERR entered; Req 6.2 → "E0" rendered.)_
5. **Confirm commanded speed forced to 0.** Verify the firmware is not commanding any nonzero
   speed (belt does not run / commanded speed reads 0). _(Req 6.3.)_
6. **Confirm continuous downstream emergency-stop transmission.** On the UART1 TX line,
   observe repeated emergency-stop frames `0xAA 0x55` being transmitted while "E0" is
   displayed. _(Req 6.3.)_
7. **Leave the board running** for at least 10 s and confirm "E0" remains latched and the
   emergency-stop frames continue (the timeout condition stays satisfied while UART1 is
   silent).

## Contrast with task 7.1 (bench build)

Task 7.1 flashes the **bench** build (`TEST_BYPASS_COMM_ERR` defined) on the **same board with
the same "no controller connected" wiring** and confirms the display shows **NO "E0"** and no
periodic 800 ms COMM_ERR beep after 2 s.

| Aspect | 7.1 Bench build (macro defined) | 7.2 Production build (macro undefined) |
|--------|--------------------------------|----------------------------------------|
| UART1 controller | not connected | not connected (identical) |
| After ~2 s | **no "E0"**, no 800 ms beep | **"E0"** shown |
| Commanded speed | not forced to 0 by timeout | forced to 0 |
| UART1 downstream | normal state-appropriate frames | continuous `0xAA55` emergency-stop |
| E17 (PA4) / E2 (overheat) | active | active |

Because the wiring and board are identical between 7.1 and 7.2 and only the compile-time
macro differs, the divergence at ~2 s (E0 vs. no E0) proves the two configurations differ
**only in the E0 timeout path as intended** (Requirement 6, and by contrast Requirement 2).
